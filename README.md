# FitFindr

FitFindr is a multi-tool AI agent that helps users find secondhand clothing and style it by orchestrating three specific tools in response to natural language queries.

## Tool Inventory

1. **`search_listings(description: str, size: str | None, max_price: float | None) -> list[dict]`**
   * **Purpose:** Searches the mock listings dataset and returns matching items based on user criteria. 
   * **Inputs:** `description` (keywords to match against listing fields), `size` (specific size string to filter by), `max_price` (maximum price limit).
   * **Outputs:** A list of dictionaries representing the matching listings.

2. **`suggest_outfit(new_item: dict, wardrobe: dict) -> str`**
   * **Purpose:** Uses an LLM (Groq's llama-3.3-70b-versatile) to suggest one or more complete outfit combinations based on a new item and the user's existing closet.
   * **Inputs:** `new_item` (the specific item dictionary selected from the search), `wardrobe` (the user's current wardrobe data structure).
   * **Outputs:** A natural language string describing how to style the new item.

3. **`create_fit_card(outfit: str, new_item: dict) -> str`**
   * **Purpose:** Generates a short, shareable Instagram-style caption of the complete outfit.
   * **Inputs:** `outfit` (the outfit suggestion generated by the previous tool), `new_item` (the dictionary containing the details of the selected item).
   * **Outputs:** A short, catchy caption string.

## The Planning Loop
The agent's logic flows sequentially but uses conditional branching to handle state changes:
1. It uses regular expressions to parse the user's natural language query and extract the `max_price`, `size`, and `description`.
2. It passes these parsed parameters to `search_listings`.
3. **Conditional Logic:** It checks the length of the search results. If the results list is empty, the loop halts immediately, sets an error string, and returns early to prevent passing null data to the LLM tools.
4. If results exist, it selects the top item and passes it to `suggest_outfit`.
5. It takes the returned outfit string and passes it into `create_fit_card`, finally returning the fully populated session to the user interface.

## State Management
State is managed via a single `session` dictionary that acts as the source of truth for a single interaction. Data is stored directly into the dictionary at each step (e.g., `session["search_results"]`, `session["selected_item"]`, `session["outfit_suggestion"]`). This data is manually passed into subsequent tool calls as parameters (e.g., passing `session["selected_item"]` into the `suggest_outfit` tool), ensuring state flows predictably from one function to the next.

## Error Handling Strategy
Every tool handles its own failure mode gracefully by returning an informative string or safe default rather than throwing a Python exception. 
* **`search_listings`:** If no items match, it returns an empty list. The planning loop catches this and tells the user: *"I couldn't find any items matching that description..."* (Tested with: `"designer ballgown size XXS under $5"`).
* **`suggest_outfit`:** If the user selects the "Empty wardrobe" option, the tool detects that `wardrobe['items']` is empty and falls back to providing general styling advice for the item rather than crashing.
* **`create_fit_card`:** Guards against receiving an empty string and returns a safe fallback message if triggered.

## Spec Reflection
* **How it helped:** Defining the exact inputs, outputs, and failure modes in the `planning.md` file made implementing the Python functions incredibly fast. We knew exactly what parameters to pass and what type to expect back.
* **How it diverged:** During implementation, we added Regex parsing to the planning loop to intelligently extract the price and size from the natural language query, rather than relying on a basic string split. This made the search tool significantly more accurate.

## AI Usage
1. **Tool Implementation:** I provided an AI tool with my exact spec definitions from `planning.md` and asked it to generate the three Python functions in `tools.py`.
2. **Debugging:** When wiring up the LLM, the generated code caused a Python namespace collision (`module 'xmlrpc.client' has no attribute 'chat'`). I directed the AI to revise the code to use the specific `_get_groq_client()` helper function provided in the starter code, which fixed the crash.
